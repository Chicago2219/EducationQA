# Lesson 3. Практика: SQL для тестировщика

## Цель практики

Имитация реальных задач, с которыми junior QA сталкивается при работе с базой данных.

После выполнения всех заданий ученик должен уметь:

- писать запросы SELECT для проверки данных;
- находить дефекты данных напрямую в базе;
- проверять бизнес-логику через SQL;
- объяснять, почему найденная проблема является дефектом.

---

## Как выполнять практику

Часть задач решается на тренажёрах:

- **[SQL Academy](https://sql-academy.org/ru/trainer)** — задачи на русском, интерактивная среда, удобна для старта.
- **[SQL-ex.ru](https://sql-ex.ru/)** — более сложные задачи, нужна регистрация, ближе к реальным кейсам.

Для каждой задачи нужно написать не только решение, но и объяснить: **зачем тестировщику уметь это делать**.

Зайди на [sql-academy.org/ru/trainer](https://sql-academy.org/ru/trainer)

### Задача 1.1 — Базовый SELECT

**Условие:** Выведи все рейсы из таблицы `Trip`.

**Решение:**

```sql
SELECT * FROM Trip;
```

**Зачем это тестировщику:** самый базовый запрос для проверки наличия данных в таблице. Первое, что делает QA, когда нужно убедиться, что данные вообще есть.

---

### Задача 1.2 — Фильтрация данных

**Условие:** Найди все рейсы, которые летят из Москвы (`Moscow`).

**Решение:**

```sql
SELECT * FROM Trip
WHERE town_from = 'Moscow';
```

**Зачем это тестировщику:** проверка, что фильтр по городу работает корректно — аналог теста фильтрации в UI.

---

### Задача 1.3 — Сортировка

**Условие:** Выведи всех пассажиров из таблицы `Passenger`, отсортированных по имени в алфавитном порядке.

**Решение:**

```sql
SELECT * FROM Passenger
ORDER BY name ASC;
```

**Зачем это тестировщику:** проверка корректности сортировки. Если в UI список отображается не в том порядке, этот запрос поможет понять, какой порядок «правильный» по данным.

---

### Задача 1.4 — Подсчёт записей

**Условие:** Сколько всего пассажиров в базе?

**Решение:**

```sql
SELECT COUNT(*) AS total_passengers FROM Passenger;
```

**Зачем это тестировщику:** проверка пагинации. Если UI говорит «Найдено 150 результатов», а база вернула 148 — это дефект.

---

### Задача 1.5 — Поиск дубликатов

**Условие:** Найди пассажиров с одинаковыми именами (дубликаты по полю `name`).

**Решение:**

```sql
SELECT name, COUNT(*) AS count
FROM Passenger
GROUP BY name
HAVING COUNT(*) > 1;
```

**Зачем это тестировщику:** дубликаты — частый класс дефектов. Например, двойной клик по кнопке регистрации может создать двух пользователей с одним email. Этот запрос находит такие случаи напрямую в базе.

---

### Задача 1.6 — NULL в данных

**Условие:** Найди все записи в таблице `Trip`, где поле `plane` не заполнено (IS NULL).

**Решение:**

```sql
SELECT * FROM Trip
WHERE plane IS NULL;
```

**Зачем это тестировщику:** NULL в обязательном поле — дефект. QA проверяет, что сервер не сохраняет запись без обязательных данных.

---

### Задача 1.7 — Агрегация и группировка

**Условие:** Выведи количество рейсов по каждому городу отправления.

**Решение:**

```sql
SELECT town_from, COUNT(*) AS trips_count
FROM Trip
GROUP BY town_from
ORDER BY trips_count DESC;
```

**Зачем это тестировщику:** если в UI раздел «Статистика по городам» показывает неверные числа, этот запрос даёт правильные значения для сравнения.

---

### Задача 1.8 — JOIN между таблицами

**Условие:** Выведи имена пассажиров и названия рейсов, которыми они летели.

**Решение:**

```sql
SELECT p.name AS passenger_name, t.id AS trip_id, t.town_from, t.town_to
FROM Passenger p
INNER JOIN Pass_in_trip pit ON p.id = pit.passenger
INNER JOIN Trip t ON pit.trip = t.id;
```

**Зачем это тестировщику:** проверка корректности связей между сущностями. Если у пассажира в UI показывается не тот рейс — проблема в маппинге данных или в JOIN-логике на бэкенде.

---

### Задача 1.9 — LEFT JOIN для поиска пропущенных данных

**Условие:** Найди пассажиров, у которых нет ни одного рейса.

**Решение:**

```sql
SELECT p.name
FROM Passenger p
LEFT JOIN Pass_in_trip pit ON p.id = pit.passenger
WHERE pit.passenger IS NULL;
```

**Зачем это тестировщику:** проверка «осиротевших» записей. Если пользователь создан, но у него нет связанных данных там, где они должны быть — это повод для расследования.

---

### Задача 1.10 — Диапазон значений (BETWEEN)

**Условие:** Найди все рейсы, у которых время отправления (`time_out`) между `10:00:00` и `15:00:00`.

**Решение:**

```sql
SELECT * FROM Trip
WHERE time_out BETWEEN '10:00:00' AND '15:00:00'
ORDER BY time_out;
```

**Зачем это тестировщику:** тестирование фильтра по времени или диапазону дат. Граничные значения (`10:00:00` и `15:00:00`) должны входить в результат — важная проверка для тестирования boundary values.

### Задача 2.1 — Задание #1 на sql-ex.ru

**Условие:** Найдите номер модели, скорость и размер жёсткого диска для всех ПК стоимостью менее 500 долларов.

**Решение:**

```sql
SELECT model, speed, hd
FROM PC
WHERE price < 500;
```

**Разбор:**

- `SELECT model, speed, hd` — выбираем только нужные поля, не все;
- `WHERE price < 500` — фильтр по условию.

**Зачем это тестировщику:** аналог проверки фильтрации по цене в каталоге. Если UI показывает товары дороже указанного порога — это баг.

---

### Задача 2.2 — Задание #2 на sql-ex.ru

**Условие:** Найдите производителей принтеров. Вывести: maker.

**Решение:**

```sql
SELECT DISTINCT maker
FROM Product
WHERE type = 'Printer';
```

**Разбор:**

- `DISTINCT` — убирает дубликаты из результата;
- `WHERE type = 'Printer'` — фильтр по типу.

**Зачем это тестировщику:** проверка уникальности в выпадающем списке. Если в UI фильтр «Производитель» показывает одного производителя дважды — дефект отображения.

---

### Задача 2.3 — Задание #3 на sql-ex.ru

**Условие:** Найдите номер модели, объём памяти и размеры экранов ПК-блокнотов, цена которых превышает 1000 долларов.

**Решение:**

```sql
SELECT model, ram, screen
FROM Laptop
WHERE price > 1000;
```

**Разбор:** базовый SELECT с условием по числовому полю.

**Зачем это тестировщику:** сверить данные в UI с данными в базе для конкретной ценовой категории.

---

### Задача 2.4 — Задание #4 на sql-ex.ru

**Условие:** Найдите всех производителей, выпускающих ПК с объёмом памяти 512 Мб и более.

**Решение:**

```sql
SELECT DISTINCT maker
FROM Product
WHERE model IN (
    SELECT model FROM PC WHERE ram >= 512
);
```

**Разбор:**

- Подзапрос (subquery) — сначала находим модели ПК с нужным объёмом RAM;
- Внешний запрос — находим производителей этих моделей;
- `DISTINCT` — убираем дубликаты.

**Зачем это тестировщику:** проверка сложных фильтров, затрагивающих несколько таблиц. Если UI-фильтр «Производители с RAM ≥ 512» показывает неправильный список — подзапрос поможет найти правильный ответ.

---

### Задача 2.5 — Задание #5 на sql-ex.ru

**Условие:** Найдите номер модели и цену каждого продукта (любого типа) производителя B.

**Решение:**

```sql
SELECT model, price FROM PC
WHERE model IN (SELECT model FROM Product WHERE maker = 'B')
UNION
SELECT model, price FROM Laptop
WHERE model IN (SELECT model FROM Product WHERE maker = 'B')
UNION
SELECT model, price FROM Printer
WHERE model IN (SELECT model FROM Product WHERE maker = 'B');
```

**Разбор:**

- `UNION` объединяет результаты из трёх таблиц в один список;
- `UNION` (без `ALL`) автоматически убирает дубликаты между таблицами.

**Зачем это тестировщику:** если раздел «Все товары производителя» в UI показывает неполный список — возможно, фронтенд делает запрос только к одной таблице.

---

### Задача 2.6 — Задание #14 на sql-ex.ru

**Условие:** Найдите класс, имя и страну для кораблей, потопленных в сражении у Гвадалканала.

*(Схема: `Ships(name, class, launched)`, `Outcomes(ship, battle, result)`, `Battles(name, date)`)*

**Решение:**

```sql
SELECT s.class, s.name, c.country
FROM Ships s
JOIN Classes c ON s.class = c.class
JOIN Outcomes o ON s.name = o.ship
WHERE o.battle = 'Guadalcanal' AND o.result = 'sunk';
```

**Разбор:**

- Соединяем три таблицы через JOIN;
- Фильтруем по имени битвы и результату.

**Зачем это тестировщику:** JOIN между несколькими таблицами — частый сценарий при проверке связанных сущностей. Например: найти все заказы пользователя, у которых статус «отменён».

---

### Задача 2.7 — Агрегация с HAVING

**Условие (аналог задания #18):** Найдите производителей, выпускающих более одной модели ПК.

**Решение:**

```sql
SELECT maker, COUNT(model) AS model_count
FROM Product
WHERE type = 'PC'
GROUP BY maker
HAVING COUNT(model) > 1;
```

**Разбор:**

- `GROUP BY` группирует по производителю;
- `HAVING` фильтрует группы (в отличие от `WHERE`, который фильтрует строки).

**Зачем это тестировщику:** проверка агрегированных данных в UI — например, счётчики в разделе «Статистика».

## Часть 3. Задачи в контексте тестирования

Эти задачи имитируют реальные рабочие задачи QA.

Используй следующую учебную схему (можно создать в PostgreSQL локально):

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    first_name VARCHAR(100),
    last_name VARCHAR(100),
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    category VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    total_amount DECIMAL(10,2),
    status VARCHAR(50) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT NOW()
);
```

---

### Задача 3.1 — Проверка последней созданной записи

**Сценарий:** Пользователь зарегистрировался. Нужно убедиться, что запись появилась в базе.

**Запрос:**

```sql
SELECT * FROM users
ORDER BY created_at DESC
LIMIT 1;
```

**Что проверяем:**

- запись появилась;
- email соответствует введённому;
- `is_active = TRUE`;
- `created_at` — сегодняшняя дата.

---

### Задача 3.2 — Поиск дубликатов email

**Сценарий:** Подозрение, что двойной клик на кнопку «Зарегистрироваться» создал двух пользователей с одним email.

**Запрос:**

```sql
SELECT email, COUNT(*) AS count
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

**Если запрос вернул строки — это дефект.** Оформляем баг-репорт: дублирующиеся записи, нарушение уникальности email.

---

### Задача 3.3 — NULL в обязательных полях

**Сценарий:** QA подозревает, что некоторые пользователи сохранились без имени.

**Запрос:**

```sql
SELECT * FROM users
WHERE first_name IS NULL OR last_name IS NULL;
```

**Если есть результаты:** либо фронтенд отправляет форму без валидации, либо бэкенд не проверяет обязательные поля.

---

### Задача 3.4 — Проверка корректности цены

**Сценарий:** Проверить, что все товары имеют цену больше нуля.

**Запрос:**

```sql
SELECT * FROM products
WHERE price <= 0;
```

**Если есть товары с ценой 0 или отрицательной — это дефект данных.**

---

### Задача 3.5 — Проверка статуса заказа

**Сценарий:** После отмены заказа в UI — убедиться, что статус обновился в базе.

**Запрос:**

```sql
SELECT id, status, created_at
FROM orders
WHERE id = 42;
```

Ожидаемый результат: `status = 'cancelled'`.

---

### Задача 3.6 — Заказы без пользователя (orphaned records)

**Сценарий:** Проверить, нет ли заказов, у которых нет связанного пользователя.

**Запрос:**

```sql
SELECT o.*
FROM orders o
LEFT JOIN users u ON o.user_id = u.id
WHERE u.id IS NULL;
```

**Если есть результаты:** нарушение целостности данных — серьёзный дефект.

---

### Задача 3.7 — Проверка уникальности email (негативный тест)

**Сценарий:** Попытаться вставить пользователя с уже существующим email и убедиться, что база даёт ошибку.

```sql
INSERT INTO users (email, first_name, last_name)
VALUES ('ivan@mail.ru', 'Иван2', 'Тестовый');
```

**Ожидаемый результат:** `ERROR: duplicate key value violates unique constraint "users_email_key"`.

**Если ошибки нет — это дефект:** поле `email` не защищено ограничением `UNIQUE`, возможны дубликаты.

---

### Задача 3.8 — Статистика по статусам заказов

**Сценарий:** В разделе аналитики UI показывает статистику по заказам. Проверить корректность цифр.

**Запрос:**

```sql
SELECT status, COUNT(*) AS count
FROM orders
GROUP BY status
ORDER BY count DESC;
```

**Сравниваем результат запроса с тем, что показывает UI.** Расхождение — дефект.

## Часть 4. Баг-репорты по SQL-находкам

После выполнения задач из Части 3 оформь 2 баг-репорта по шаблону.

### Шаблон баг-репорта

```
Summary:
Environment:
Preconditions:
Steps to reproduce:
  1.
  2.
  3.
Actual result:
Expected result:
Severity:
Priority:
Attachments / Notes:
```

### Пример баг-репорта 1

**Summary:** При двойном клике на кнопку «Зарегистрироваться» создаются дубликаты пользователя в базе данных

**Environment:** Windows 11, Chrome 124, staging-стенд, PostgreSQL 15

**Preconditions:** В базе нет пользователя с email `test@mail.ru`

**Steps to reproduce:**
1. Открыть страницу регистрации.
2. Заполнить форму: email `test@mail.ru`, имя, пароль.
3. Дважды быстро кликнуть по кнопке «Зарегистрироваться».
4. Выполнить SQL-запрос: `SELECT email, COUNT(*) FROM users WHERE email = 'test@mail.ru' GROUP BY email;`

**Actual result:** Запрос возвращает `count = 2`. В базе две записи с одним email.

**Expected result:** Запрос возвращает `count = 1`. Второй клик не создаёт дубликат.

**Severity:** Major

**Priority:** High

**Attachments / Notes:** Скриншот результата SQL-запроса. Скриншот Network с двумя одинаковыми POST-запросами на `/api/register`.

---

### Пример баг-репорта 2

**Summary:** Товары с ценой 0 присутствуют в базе данных и отображаются в каталоге

**Environment:** Windows 11, Chrome 124, staging-стенд

**Preconditions:** Открыт каталог товаров

**Steps to reproduce:**
1. Открыть каталог товаров в UI.
2. Убедиться, что некоторые товары отображаются с ценой «0».
3. Выполнить SQL-запрос: `SELECT id, title, price FROM products WHERE price <= 0;`

**Actual result:** Запрос возвращает 3 товара с ценой 0. В UI они отображаются в каталоге без предупреждения.

**Expected result:** Товары с ценой 0 не должны отображаться в публичном каталоге. SQL-запрос не должен возвращать результатов.

**Severity:** Major

**Priority:** High

**Attachments / Notes:** Результат SQL-запроса, скриншоты карточек товаров с ценой 0.

## Критерии хорошего выполнения практики

Практика выполнена хорошо, если ученик:

- решил все задачи и объяснил, зачем каждая нужна тестировщику;
- умеет находить дубликаты, NULL-значения и некорректные данные через SQL;
- умеет использовать JOIN для проверки связей между сущностями;
- оформил баг-репорты с SQL-запросами как доказательством дефекта;
- понимает разницу между `WHERE` и `HAVING`;
- не делает UPDATE/DELETE без `WHERE`.
