## 1. Зачем тестировщику SQL

На реальном проекте QA постоянно нужно:

- проверить, что данные из формы действительно сохранились в базе;
- убедиться, что при удалении записи в UI она удалена и в базе;
- найти тестовые данные для конкретного сценария;
- убедиться, что данные не дублируются;
- проверить, что сервер не сохраняет лишнее;
- сверить данные в UI с данными в базе.

**Пример реальной задачи:**

Пользователь заполнил форму регистрации. В UI появилось сообщение «Регистрация успешна». QA должен убедиться, что запись появилась в таблице `users` с корректными данными.

Без SQL тестировщик верит UI на слово. С SQL — проверяет напрямую.

## 2. Реляционная модель данных

Реляционная база данных — хранилище данных, организованное в виде таблиц.

### 2.1 Таблица (Table / Relation)

Таблица — структура из строк и столбцов.

Пример таблицы `users`:

| id | email | first_name | last_name | created_at |
|---|---|---|---|---|
| 1 | ivan@mail.ru | Иван | Петров | 2024-01-15 |
| 2 | anna@mail.ru | Анна | Сидорова | 2024-01-16 |
| 3 | mike@mail.ru | Михаил | Козлов | 2024-01-17 |

### 2.2 Основные термины

| Термин | Описание |
|---|---|
| Строка (Row / Record) | Одна запись в таблице. В примере — один пользователь |
| Столбец (Column / Attribute) | Отдельное поле: `id`, `email`, `first_name` |
| Первичный ключ (PK) | Уникальный идентификатор строки. Не NULL, не повторяется |
| Внешний ключ (FK) | Поле, ссылающееся на PK другой таблицы |
| NULL | Отсутствие значения. Не то же самое, что `0` или `''` |

### 2.3 Связи между таблицами

| Тип связи | Описание | Пример |
|---|---|---|
| One-to-One (1:1) | Один к одному | Пользователь — один профиль |
| One-to-Many (1:N) | Один ко многим | Пользователь — много заказов |
| Many-to-Many (N:M) | Многие ко многим | Товары и заказы |

```
users.id  ←──  orders.user_id
```

### 2.4 Почему NULL важен для QA

- `email = NULL` и `email = ''` — разные состояния;
- если поле обязательное, NULL — это дефект;
- NULL влияет на сортировку, агрегацию, JOIN;
- `COUNT(email)` не считает NULL, а `COUNT(*)` считает все строки.

## 3. Типы данных в PostgreSQL

Понимание типов данных критично: нужно знать, какие значения корректны, а какие нет.

### 3.1 Числовые типы

| Тип | Диапазон | Применение |
|---|---|---|
| `SMALLINT` | -32 768 до 32 767 | Возраст, небольшие счётчики |
| `INTEGER` (INT) | -2 147 483 648 до 2 147 483 647 | ID, количества |
| `BIGINT` | ±9,2 × 10^18 | Большие ID, транзакции |
| `DECIMAL(p,s)` / `NUMERIC` | Точное дробное | **Цены, финансы** |
| `REAL` | ~6 знаков после запятой | Координаты (не деньги!) |
| `DOUBLE PRECISION` | ~15 знаков после запятой | Научные расчёты |

**Важно для QA:** цена товара должна храниться как `DECIMAL`, а не `REAL`. Числа с плавающей точкой дают погрешность: `109.95` может стать `109.9499999`. Если видишь баг с ценой — проверь тип поля в схеме.

---

### 3.2 Строковые типы

| Тип | Описание | Применение |
|---|---|---|
| `CHAR(n)` | Фиксированная длина n | Коды стран: `RU`, `US` |
| `VARCHAR(n)` | До n символов | Имена, email, заголовки |
| `TEXT` | Без ограничения | Описания, комментарии |

**Важно для QA:**

- `VARCHAR(50)` — ввод 51 символа должен давать ошибку;
- граничные значения: 0 символов, ровно n, n+1;
- `CHAR(10)` дополняет строку пробелами — неожиданное поведение при поиске.

---

### 3.3 Логический тип

| Тип | Значения |
|---|---|
| `BOOLEAN` | `TRUE`, `FALSE`, `NULL` |

**Важно для QA:** `is_active = NULL` — это не `FALSE`. Это «неизвестно». Фильтр `WHERE is_active = TRUE` исключит и `FALSE`, и `NULL`.

---

### 3.4 Типы дат и времени

| Тип | Описание | Пример |
|---|---|---|
| `DATE` | Только дата | `2024-01-15` |
| `TIME` | Только время | `14:30:00` |
| `TIMESTAMP` | Дата и время без часового пояса | `2024-01-15 14:30:00` |
| `TIMESTAMPTZ` | Дата и время с часовым поясом | `2024-01-15 14:30:00+03` |
| `INTERVAL` | Промежуток | `3 days`, `2 hours` |

**Важно для QA:**

- `TIMESTAMP` без пояса — проблема для международных пользователей;
- граничные даты: `2024-02-29` (существует только в високосный год);
- дата рождения не может быть в будущем.

---

### 3.5 Специальные типы PostgreSQL

| Тип | Описание | Применение |
|---|---|---|
| `UUID` | Уникальный идентификатор | ID пользователей, токены |
| `JSON` / `JSONB` | JSON-данные | Гибкие данные, настройки |
| `ARRAY` | Массив значений | Теги, роли |
| `ENUM` | Перечисление значений | Статусы, типы |

Пример `ENUM`:

```sql
CREATE TYPE order_status AS ENUM ('pending', 'processing', 'shipped', 'delivered', 'cancelled');
```

**Важно для QA:** если поле `status` — `ENUM`, попытка записать туда значение вне списка должна вызывать ошибку — отличный негативный тест.

## 4. Мощность алфавита и кодировка UTF-8

### 4.1 Что такое алфавит в информатике

**Алфавит** — конечное непустое множество символов, из которых составляются строки.

**Мощность алфавита** — количество символов в алфавите.

Примеры:

| Алфавит | Мощность |
|---|---|
| Двоичный: `{0, 1}` | 2 |
| Десятичные цифры: `{0–9}` | 10 |
| Латиница (строчные): `{a–z}` | 26 |
| ASCII | 128 |
| Unicode 15.1 | более 149 000 |

Из алфавита мощностью `A`, строки длиной `n` можно составить **A^n** вариантов.

Примеры:

- Двоичный, длина 3: `2^3 = 8` (`000`, `001`, `010`…)
- Латиница, длина 2: `26^2 = 676`

**Почему важно для QA:**

- Надёжность паролей — чем больше алфавит и длина, тем больше вариантов.
- `VARCHAR(255)` в UTF-8 может занять до `255 × 4 = 1020` байт (не 255 байт!).
- Помогает оценивать граничные сценарии ввода данных.

---

### 4.2 ASCII

**ASCII (American Standard Code for Information Interchange)** — стандарт 1963 года.

- 7 бит → **128 символов** (0–127)
- Включает: латиницу, цифры, знаки препинания, управляющие символы
- Не включает: кириллицу, иероглифы, эмодзи

Примеры ASCII-кодов:

| Символ | Десятичный | Шестнадцатеричный |
|---|---|---|
| A | 65 | 0x41 |
| a | 97 | 0x61 |
| 0 | 48 | 0x30 |
| Пробел | 32 | 0x20 |
| ! | 33 | 0x21 |

---

### 4.3 Unicode

**Unicode** — универсальный стандарт, охватывающий символы всех языков мира.

- Unicode 15.1: более **149 000 символов**
- Каждый символ имеет уникальный **code point**: `U+0041` = `A`, `U+1F600` = 😀
- Охватывает кириллицу, иероглифы, арабский, эмодзи и многое другое.

---

### 4.4 UTF-8

**UTF-8 (Unicode Transformation Format, 8-bit)** — наиболее популярная кодировка Unicode.

Особенности:

- Переменная длина: от **1 до 4 байт** на символ
- Обратно совместима с ASCII (первые 128 символов = 1 байт)
- Используется в большинстве баз данных, API и веб-сервисов

#### Таблица кодировки UTF-8

| Диапазон Unicode | Байт | Шаблон | Примеры |
|---|---|---|---|
| U+0000 – U+007F | 1 | `0xxxxxxx` | ASCII: A, 0, ! |
| U+0080 – U+07FF | 2 | `110xxxxx 10xxxxxx` | Кириллица: И, А, Я |
| U+0800 – U+FFFF | 3 | `1110xxxx 10xxxxxx 10xxxxxx` | Иероглифы: 中, 日 |
| U+10000 – U+10FFFF | 4 | `11110xxx 10xxxxxx 10xxxxxx 10xxxxxx` | Эмодзи: 😀, 🔥 |

#### Примеры кодирования символов

| Символ | Unicode | UTF-8 байты |
|---|---|---|
| `A` | U+0041 | `41` (1 байт) |
| `И` | U+0418 | `D0 98` (2 байта) |
| `中` | U+4E2D | `E4 B8 AD` (3 байта) |
| `😀` | U+1F600 | `F0 9F 98 80` (4 байта) |

---

### 4.5 Почему кодировки важны для QA

| Сценарий | Потенциальный баг |
|---|---|
| Имя на кириллице в форме | Отображается `????????` |
| Эмодзи в поле имени | Ошибка БД (если нет поддержки 4-байтовых символов) |
| Текст с кавычками `"` | SQL-инъекция или ошибка парсинга |
| Арабский текст | RTL ломает вёрстку |
| CSV-экспорт на Windows | Windows-1251 вместо UTF-8 — кириллица ломается |

Что проверять:

- поле принимает кириллицу, иероглифы, арабский текст;
- эмодзи не вызывают ошибку сервера;
- данные сохраняются и отображаются без потери символов;
- экспорт в CSV открывается корректно.

## 5. Базовый SQL для тестировщика

### 5.1 SELECT — получение данных

```sql
-- Все строки
SELECT * FROM users;

-- Только нужные поля
SELECT id, email, first_name FROM users;

-- С условием
SELECT * FROM users WHERE id = 1;

-- Несколько условий
SELECT * FROM users WHERE first_name = 'Ivan' AND is_active = TRUE;

-- Поиск по части строки
SELECT * FROM users WHERE email LIKE '%@mail.ru';

-- Сортировка по убыванию
SELECT * FROM users ORDER BY created_at DESC;

-- Ограничение и пропуск
SELECT * FROM users LIMIT 10 OFFSET 20;
```

---

### 5.2 Операторы WHERE

| Оператор | Пример |
|---|---|
| `=` | `WHERE id = 5` |
| `!=` / `<>` | `WHERE status != 'deleted'` |
| `>`, `<`, `>=`, `<=` | `WHERE age >= 18` |
| `BETWEEN` | `WHERE price BETWEEN 100 AND 500` |
| `IN` | `WHERE status IN ('active', 'pending')` |
| `NOT IN` | `WHERE role NOT IN ('admin')` |
| `LIKE` | `WHERE email LIKE '%@gmail.com'` |
| `IS NULL` | `WHERE deleted_at IS NULL` |
| `IS NOT NULL` | `WHERE email IS NOT NULL` |
| `AND` / `OR` / `NOT` | `WHERE age > 18 AND is_active = TRUE` |

---

### 5.3 INSERT — добавление данных

```sql
INSERT INTO users (email, first_name, last_name)
VALUES ('test@mail.ru', 'Тест', 'Тестов');
```

Применение: создать тестовую запись напрямую в базе, чтобы проверить отображение в UI.

---

### 5.4 UPDATE — обновление данных

```sql
UPDATE users
SET email = 'new@mail.ru'
WHERE id = 5;

-- ВНИМАНИЕ: без WHERE обновятся ВСЕ строки!
```

Применение: изменить статус записи, чтобы проверить поведение UI в разных состояниях.

---

### 5.5 DELETE — удаление данных

```sql
DELETE FROM users WHERE id = 5;

-- ВНИМАНИЕ: без WHERE удалятся ВСЕ строки!
```

**Правило:** перед UPDATE и DELETE — сначала делай `SELECT` с тем же `WHERE`, чтобы видеть, что будет затронуто.

---

### 5.6 Агрегатные функции

```sql
SELECT COUNT(*) FROM users;
SELECT COUNT(email) FROM users;          -- не считает NULL
SELECT SUM(price) FROM orders WHERE user_id = 1;
SELECT AVG(price) FROM products;
SELECT MAX(price), MIN(price) FROM products;
```

---

### 5.7 GROUP BY

```sql
-- Количество пользователей по статусу
SELECT status, COUNT(*) AS count
FROM users
GROUP BY status;

-- Сумма заказов по пользователям
SELECT user_id, SUM(total_amount) AS total
FROM orders
GROUP BY user_id
ORDER BY total DESC;
```

---

### 5.8 JOIN — соединение таблиц

```sql
-- INNER JOIN: только совпадения
SELECT u.first_name, u.email, o.id AS order_id, o.total_amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: все из левой + совпадения из правой
SELECT u.first_name, o.id AS order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
-- Пользователи без заказов — с NULL в полях orders
```

---

### 5.9 Полезные запросы для тестировщика

```sql
-- Последняя созданная запись
SELECT * FROM users ORDER BY created_at DESC LIMIT 1;

-- Найти дубликаты email
SELECT email, COUNT(*) AS count
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- Поля с NULL
SELECT * FROM users WHERE email IS NULL OR first_name IS NULL;

-- Некорректные цены
SELECT * FROM products WHERE price <= 0;

-- Все заказы пользователя
SELECT * FROM orders WHERE user_id = 5 ORDER BY created_at DESC;
```

## 6. Схема базы данных

QA читает схему, чтобы понять ограничения и риски.

```
users
  id         BIGINT         PK
  email      VARCHAR(255)   NOT NULL UNIQUE
  first_name VARCHAR(100)   NOT NULL
  last_name  VARCHAR(100)   NOT NULL
  is_active  BOOLEAN        DEFAULT TRUE
  created_at TIMESTAMP      DEFAULT NOW()

products
  id          BIGINT         PK
  title       VARCHAR(255)   NOT NULL
  price       DECIMAL(10,2)  NOT NULL
  category_id INTEGER        FK → categories.id
  created_at  TIMESTAMP      DEFAULT NOW()

orders
  id           BIGINT         PK
  user_id      BIGINT         FK → users.id
  total_amount DECIMAL(10,2)  NOT NULL
  status       VARCHAR(50)    DEFAULT 'pending'
  created_at   TIMESTAMP      DEFAULT NOW()

order_items
  id         BIGINT         PK
  order_id   BIGINT         FK → orders.id
  product_id BIGINT         FK → products.id
  quantity   INTEGER        NOT NULL
  price      DECIMAL(10,2)  NOT NULL
```

Что QA видит из схемы:

- `email` — `NOT NULL UNIQUE`: пустой и дублирующийся должны давать ошибку;
- `price DECIMAL(10,2)`: максимум 8 цифр до запятой и 2 после;
- `status VARCHAR(50)` без `ENUM`: сервер не ограничивает значения — нужна проверка валидации в коде.

## 7. Типичные SQL-баги, которые находит QA

| Баг | Запрос для проверки |
|---|---|
| Запись не создалась | `SELECT * FROM users ORDER BY created_at DESC LIMIT 1` |
| Дубликаты email | `SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1` |
| NULL в обязательном поле | `SELECT * FROM users WHERE first_name IS NULL` |
| Некорректная цена | `SELECT * FROM products WHERE price <= 0` |
| Статус не обновился | `SELECT status FROM orders WHERE id = X` |
| Удалённая запись не помечена | `SELECT * FROM users WHERE id = X` |
