# Lesson 1. Практика: реальные задачи junior QA

## Цель практики

Этот файл нужен не как «список абстрактных упражнений», а как имитация первых рабочих задач manual QA на стажировке или в коммерческом проекте. Логика простая: ученик получает задачу, анализирует сценарий, проводит проверки, фиксирует результат и оформляет артефакты так, как это делал бы на реальной работе.

После выполнения всех заданий из этого файла ученик должен уметь:

- анализировать функциональность до тестирования;
- выделять позитивные и негативные сценарии;
- проверять данные в `UI`, `Network` и `API`;
- работать с JSON-ответами;
- составлять чек-листы;
- писать качественные баг-репорты;
- объяснять, почему найденная проблема действительно является дефектом.

---

## Как выполнять практику

Для выполнения практики понадобятся:

- `Google Chrome`
- `DevTools`
- `Postman`
- любой текстовый редактор или `Google Docs`
- при желании таблица в `Google Sheets` для оформления чек-листов

Рекомендуемый формат работы по каждому заданию:

1. Прочитать постановку задачи.
2. Определить, что именно должна делать система.
3. Выписать основные риски.
4. Составить минимум 10 проверок.
5. Выполнить проверки руками.
6. Если найден дефект — оформить баг-репорт.
7. Если задача связана с API — сверить запрос, response и ожидаемую бизнес-логику.

---

## Правило выполнения практики

На каждом задании нужно отвечать не только **что ты проверял**, но и **почему именно это нужно проверять**.

Например, слабый ответ:

- проверил логин;
- проверил пароль;
- проверил ошибку.

Сильный ответ:

- проверил успешную авторизацию, потому что это ключевой бизнес-сценарий;
- проверил валидацию пустых полей, потому что пользователь может отправить форму без данных;
- проверил двойной клик по кнопке логина, потому что это частый источник дублирующихся запросов;
- проверил код ответа API, чтобы понять, проблема в UI или на сервере.

# Блок 1. Практика по UI: форма логина

## Сценарий как на проекте

Представь, что ты junior QA в команде, которая тестирует внутренний личный кабинет пользователя. Разработчики прислали задачу: **проверить экран логина перед релизом**.

### Что известно по требованиям

На странице есть:

- поле `Email`
- поле `Password`
- кнопка `Login`
- сообщение об ошибке при неверных данных
- успешный логин должен переводить пользователя в личный кабинет

### Задача

Составь чек-лист минимум из 20 пунктов для формы логина.

### На что обратить внимание

- корректное отображение всех элементов;
- пустые поля;
- валидный и невалидный email;
- скрытие символов пароля;
- поведение кнопки `Login`;
- обработка неверных данных;
- повторная отправка формы;
- отображение ошибки;
- состояние после неуспешной попытки;
- отправляемый запрос в `Network`.

### Пример хороших проверок

- страница логина открывается без ошибок в интерфейсе;
- присутствуют оба поля и кнопка отправки;
- можно ввести корректный email;
- нельзя случайно увидеть пароль в открытом виде, если не предусмотрен toggle;
- при пустом email форма валидируется;
- при пустом пароле форма валидируется;
- при пустых обоих полях форма не должна считаться успешной;
- email с пробелами в начале и в конце корректно обрабатывается;
- email без символа `@` не принимается;
- пароль из одного символа обрабатывается корректно по требованиям;
- при неверных данных показывается сообщение об ошибке;
- после неуспешного логина форма остается работоспособной;
- по двойному клику не уходит два одинаковых запроса, если этого не должно быть;
- при успешном логине пользователь попадает в нужный раздел;
- в `Network` отправляется запрос на нужный endpoint;
- метод запроса соответствует ожиданию, обычно `POST`;
- код ответа соответствует сценарию;
- ошибка не выглядит как технический stack trace для пользователя;
- поля и кнопки визуально не ломаются при длинных значениях;
- после обновления страницы поведение остается корректным.

### Что должен сдать ученик

- чек-лист на 20+ проверок;
- 2 вымышленных, но реалистичных баг-репорта;
- краткое объяснение, какие сценарии наиболее критичны и почему.

# Блок 2. Практика по DevTools: поиск причины проблемы

## Сценарий как на проекте

Менеджер сообщает: **«На странице профиля пользователя иногда не отображается email. Нужно понять, это баг фронтенда или бэкенда»**.

### Учебная модель

Используй публичный API:

`GET https://reqres.in/api/users/2`

Ожидаемый JSON:

```json
{
  "data": {
    "id": 2,
    "email": "janet.weaver@reqres.in",
    "first_name": "Janet",
    "last_name": "Weaver",
    "avatar": "https://reqres.in/img/faces/2-image.jpg"
  }
}
```

### Задача

Представь, что в интерфейсе профиля показываются:

- имя
- фамилия
- email
- аватар

Но в UI email пустой.

### Что нужно сделать

1. Отправить запрос в Postman.
2. Убедиться, что в response поле `email` есть.
3. Представить, что на UI email не отображается.
4. Написать краткий вывод:
   - что пришло с сервера;
   - что не отобразилось в UI;
   - на какую сторону вероятнее всего заводить дефект.

### Ожидаемый вывод сильного junior QA

- API возвращает поле `email` и значение не пустое;
- проблема воспроизводится только на UI;
- вероятная зона дефекта — фронтенд, маппинг ответа в компонент профиля или логика рендера;
- в баг-репорт нужно приложить скрин UI и response из API.

### Что должен сдать ученик

- краткий анализ инцидента;
- 1 баг-репорт;
- список из 5 возможных технических причин проблемы.

### Возможные причины

- поле не маппится из response в компонент;
- используется неверное имя поля, например `mail` вместо `email`;
- email скрывается из-за условия отображения;
- данные затираются после повторного рендера;
- UI ожидает данные в другом вложенном объекте.

# Блок 3. Практика по API: проверка пользователя

## Сценарий как на проекте

Тебе дали задачу проверить endpoint профиля пользователя после обновления backend-сервиса.

Используй API:

`GET https://dummyjson.com/users/1`

### Пример полей ответа

```json
{
  "id": 1,
  "firstName": "Emily",
  "lastName": "Johnson",
  "age": 28,
  "gender": "female",
  "email": "emily.johnson@x.dummyjson.com",
  "phone": "+1 947-890-8238",
  "username": "emilys",
  "birthDate": "1996-5-30",
  "image": "https://dummyjson.com/icon/emilys/128"
}
```

### Задача

Проверь endpoint как manual QA.

### Нужно выполнить

- проверить статус-код;
- проверить, что ответ приходит в JSON;
- проверить наличие обязательных полей;
- определить, какие поля должны отображаться в профиле пользователя;
- определить, какие поля не стоит показывать в публичном UI;
- составить минимум 10 тестов для этого endpoint.

### Пример хороших тестов

- статус ответа `200`;
- `id` — число и больше 0;
- `firstName` не пустой;
- `lastName` не пустой;
- `email` содержит `@`;
- `phone` приходит строкой;
- `username` не пустой;
- `birthDate` приходит в ожидаемом формате или хотя бы не пустой;
- `image` содержит URL;
- обязательные поля не равны `null`.

### Дополнительный аналитический вопрос

Какие поля из такого ответа нельзя светить в интерфейсе без необходимости?

Ожидаемо ученик должен заметить чувствительные данные или потенциально чувствительные данные и задуматься о принципе минимально достаточного отображения данных.

### Что должен сдать ученик

- список обязательных полей;
- список чувствительных или лишних для UI полей;
- 10+ тестовых проверок;
- 1 баг-репорт на выдуманный реалистичный дефект.

# Блок 4. Практика по e-commerce: карточка товара

## Сценарий как на проекте

Ты тестируешь интернет-магазин. После обновления каталога нужно проверить карточку товара и убедиться, что данные отображаются корректно.

Используй API:

`GET https://fakestoreapi.com/products/1`

### Пример ответа

```json
{
  "id": 1,
  "title": "Fjallraven - Foldsack No. 1 Backpack, Fits 15 Laptops",
  "price": 109.95,
  "description": "Your perfect pack for everyday use and walks in the forest.",
  "category": "men's clothing",
  "image": "https://fakestoreapi.com/img/81fPKd-2AYL._AC_SL1500_.jpg",
  "rating": {
    "rate": 3.9,
    "count": 120
  }
}
```

### Задача

Представь, что на UI есть:

- название товара;
- цена;
- описание;
- категория;
- рейтинг;
- изображение.

Составь тестовый набор для проверки карточки.

### Нужно выполнить

- составить 15 позитивных и негативных проверок;
- выписать 5 рисков отображения данных;
- придумать 3 дефекта разной критичности;
- оформить 1 полноценный баг-репорт.

### Примеры рисков

- цена может отображаться как `0` или без валютного формата;
- длинное название может ломать верстку;
- картинка может не загружаться;
- рейтинг может отображаться без округления или с неверным значением;
- категория может не совпадать с данными API.

### Примеры дефектов

- **Minor**: описание обрезается без кнопки раскрытия;
- **Major**: цена отображается `0`, хотя в API корректное значение;
- **Critical**: карточка товара вообще не открывается и возвращает ошибку.

### Что должен сдать ученик

- тестовый список минимум из 15 проверок;
- таблицу рисков;
- 1 оформленный баг-репорт;
- короткое объяснение, какие проверки относятся к UI, какие к данным, а какие к интеграции UI с API.

# Блок 5. Практика по CRUD: создание записи

## Сценарий как на проекте

Команда выпускает модуль новостей. Нужно проверить создание новой записи через API, чтобы потом убедиться, что UI сможет с этим работать.

Используй API:

`POST https://jsonplaceholder.typicode.com/posts`

Body:

```json
{
  "title": "QA test post",
  "body": "Testing post creation",
  "userId": 1
}
```

### Задача

Проверить сценарий создания записи как junior QA.

### Что нужно проверить

- корректный endpoint;
- корректный метод `POST`;
- правильный формат body;
- код ответа;
- структура response;
- наличие созданного `id`;
- соответствие response отправленным данным.

### Дополнительные негативные сценарии

- пустой `title`;
- пустой `body`;
- отсутствует `userId`;
- `userId` передан строкой вместо числа;
- отправлены лишние поля.

### Что должен сдать ученик

- таблицу из 8 тестов на создание сущности;
- краткий вывод, какие ошибки относятся к валидации запроса;
- 1 баг-репорт на сценарий, где сервер принимает явно некорректные данные без ошибки.

- так быстрее понять контракт;

# Блок 12. Критерии хорошего выполнения практики

Практика выполнена хорошо, если ученик:

- не ограничивается только позитивными сценариями;
- замечает связи между UI и API;
- смотрит на данные, а не только на внешний вид страницы;
- умеет объяснить логику проверок;
- пишет воспроизводимые баг-репорты;
- может отделить проблему фронтенда от проблемы бэкенда хотя бы на базовом уровне;
- понимает, какие сценарии критичны для бизнеса.

# Блок 15. Что можно обсудить после выполнения

После выполнения практики полезно разобрать с учеником:

- где он пропустил негативные сценарии;
- насколько хорошо он выделяет критичные проверки;
- умеет ли он связывать требования, UI и API;
- насколько качественно пишет баг-репорты;

- умеет ли он рассуждать как QA, а не просто как пользователь.

Если эта практика выполнена качественно, можно переходить к следующему блоку: `HTTP`, `Network`, `Headers`, `Request/Response`, `Cookies`, `Auth`, `DevTools` уже на более глубоком уровне.
