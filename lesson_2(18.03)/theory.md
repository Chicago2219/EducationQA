# Lesson 2. Git и GitHub: теория для тестировщика

## Цель урока

После этого урока ученик должен:

- понимать, зачем тестировщику Git и GitHub;
- знать ключевые понятия: репозиторий, коммит, ветка, merge, pull request;
- уметь клонировать репозиторий и переключаться между ветками;
- читать историю изменений и диффы;
- понимать, где именно QA пересекается с Git в реальной команде;
- не бояться терминала на уровне базовых команд.

---

## 1. Зачем тестировщику Git

Многие junior QA считают, что Git — это инструмент только для разработчиков. Это распространённое заблуждение.

На реальном проекте тестировщик сталкивается с Git каждый день:

- клонирует репозиторий с тест-документацией, чек-листами, тест-кейсами;
- читает историю коммитов, чтобы понять, что именно изменилось перед релизом;
- переключается на ветку с новой фичей, чтобы протестировать её изолированно;
- смотрит pull request разработчика и оставляет комментарии;
- ведёт собственный репозиторий с артефактами тестирования;
- запускает автотесты из репозитория и читает результаты.

Минимальный набор команд для manual QA:

```
git clone
git pull
git checkout
git log
git status
git diff
```

Этого достаточно, чтобы уверенно работать в команде на первом проекте.

## 2. Основные понятия

### 2.1 Репозиторий (repository, repo)

Репозиторий — это папка с проектом, история изменений которой отслеживается Git.

| Тип | Описание |
|---|---|
| Локальный | Находится на твоём компьютере |
| Удалённый (remote) | Размещён на сервере: GitHub, GitLab, Bitbucket |

Разработчик вносит изменения локально, затем отправляет их на удалённый. QA чаще всего клонирует удалённый репозиторий и работает с ним.

---

### 2.2 Коммит (commit)

Коммит — это снимок состояния файлов в определённый момент. Каждый коммит содержит:

- автора и его email;
- дату и время;
- сообщение (commit message);
- уникальный хэш (SHA), например `a3f7c12`.

Пример хорошего коммит-сообщения:

```
fix: исправлена валидация поля email на форме логина
feat: добавлена кнопка сброса фильтров в каталоге
test: добавлены проверки для API профиля пользователя
```

Пример плохого коммит-сообщения:

```
fix
changes
update
wip
```

**Почему это важно для QA:** читая историю коммитов перед релизом, тестировщик понимает, что именно нужно проверить, и составляет фокус-область тестирования.

---

### 2.3 Ветка (branch)

Ветка — это независимая линия разработки. По умолчанию существует ветка `main` (раньше `master`).

Когда разработчик начинает новую фичу, он создаёт отдельную ветку:

```
main
 ├── feature/login-validation
 ├── feature/user-profile
 └── bugfix/price-display
```

**Почему это важно для QA:** тестировщик переключается на ветку конкретной задачи и проверяет именно те изменения, которые туда вошли, не затрагивая основную ветку.

---

### 2.4 Merge (слияние)

Merge — объединение изменений из одной ветки в другую. Когда фича готова и протестирована, ветку `feature/login-validation` вливают в `main`.

---

### 2.5 Pull Request (PR) / Merge Request (MR)

Pull Request (GitHub) или Merge Request (GitLab) — это запрос на слияние ветки. Разработчик открывает PR, когда считает, что задача готова.

В PR видно:

- какие файлы изменились;
- что именно добавилось или удалилось (дифф);
- описание изменений;
- комментарии команды.

**QA активно работает с PR:**

- читает описание, чтобы понять объём изменений;
- смотрит дифф, чтобы оценить риски;
- оставляет комментарии при нахождении проблем;
- ставит статус: Approved или Request Changes.

---

### 2.6 Clone, Pull, Push

| Команда | Что делает |
|---|---|
| `git clone <url>` | Копирует удалённый репозиторий на локальный компьютер |
| `git pull` | Скачивает последние изменения с удалённого репозитория |
| `git push` | Отправляет локальные коммиты на удалённый репозиторий |

QA чаще всего использует `clone` и `pull`. `push` нужен, если QA ведёт собственный репозиторий.

## 3. Установка и начальная настройка

### 3.1 Установка Git

- **Windows:** [git-scm.com](https://git-scm.com/)
- **macOS:** `brew install git` или встроен по умолчанию
- **Linux (Ubuntu/Debian):** `sudo apt install git`

Проверить установку:

```bash
git --version
```

### 3.2 Первоначальная настройка

```bash
git config --global user.name "Имя Фамилия"
git config --global user.email "you@example.com"
```

Эти данные будут указываться в каждом коммите как автор.

## 4. Основные команды с пояснениями

### 4.1 Клонирование репозитория

```bash
git clone https://github.com/username/repository.git
```

После выполнения появится папка с именем репозитория. Все файлы уже внутри.

---

### 4.2 Просмотр текущего состояния

```bash
git status
```

Показывает:

- текущую ветку;
- изменённые файлы, не добавленные в staging;
- файлы, готовые к коммиту.

---

### 4.3 История коммитов

```bash
git log
```

Удобный короткий формат:

```bash
git log --oneline
```

Пример вывода:

```
a3f7c12 fix: исправлен маппинг email в компоненте профиля
9b2e4f1 feat: добавлена страница профиля пользователя
3c1d8a0 init: начальная структура проекта
```

**Как QA использует:** смотрит, что менялось с момента последнего тестирования, и составляет список того, что нужно перепроверить.

Показать коммиты за последние 2 недели:

```bash
git log --oneline --since="2 weeks ago"
```

---

### 4.4 Переключение между ветками

Посмотреть все доступные ветки:

```bash
git branch -a
```

Переключиться на ветку:

```bash
git checkout feature/login-validation
```

Современный вариант (Git 2.23+):

```bash
git switch feature/login-validation
```

---

### 4.5 Получение последних изменений

```bash
git pull
```

Нужно выполнять перед тестированием, чтобы работать с актуальной версией кода.

---

### 4.6 Просмотр изменений (дифф)

```bash
git diff
```

Дифф конкретного файла:

```bash
git diff src/components/Profile.js
```

---

### 4.7 Просмотр конкретного коммита

```bash
git show a3f7c12
```

Покажет все изменения, которые вошли в этот коммит.

## 5. GitHub: что нужно знать тестировщику

GitHub — веб-платформа для хранения Git-репозиториев. Основные разделы, с которыми работает QA:

### 5.1 Code

Все файлы проекта. QA смотрит тут документацию, конфигурационные файлы, тест-скрипты.

### 5.2 Commits

История всех коммитов. Можно фильтровать по ветке, автору, дате.

Сценарий: перед регрессионным тестированием QA открывает список коммитов за период спринта, чтобы понять, что изменилось.

### 5.3 Pull Requests

Список всех открытых и закрытых PR. QA:

- читает описание задачи;
- смотрит список изменённых файлов (Files changed);
- видит дифф: **зелёный** — добавлено, **красный** — удалено;
- оставляет комментарий прямо на строке кода;
- может поставить Approve или Request changes.

### 5.4 Issues

Задачи, баги, обсуждения. QA может:

- читать и создавать задачи для баг-репортов;
- связывать PR с Issues.

### 5.5 Actions

Автоматические пайплайны (CI/CD). QA смотрит сюда, чтобы проверить, прошли ли автотесты после последнего коммита.

## 6. Чтение диффа — важный навык для QA

Когда разработчик изменяет код, дифф выглядит так:

```diff
- email: user.mail,
+ email: user.email,
```

Красная строка — что было. Зелёная строка — что стало.

Пример изменения в компоненте профиля:

```diff
- <span>{user.mail}</span>
+ <span>{user.email}</span>
```

QA, читая этот дифф, сразу понимает:

- раньше поле называлось `mail`;
- теперь оно называется `email`;
- именно поэтому email не отображался — использовалось неправильное имя поля.

Читая PR, тестировщик может предсказать, что именно нужно проверить и где искать возможные ошибки.

## 7. Как QA работает с Git на реальном проекте

### Сценарий 1: Тестирование новой фичи

Разработчик открыл PR с веткой `feature/user-avatar`. QA:

1. Открывает PR на GitHub.
2. Читает описание задачи.
3. Смотрит Files Changed — какие файлы затронуты.
4. Переключается на ветку: `git checkout feature/user-avatar`.
5. Тестирует фичу.
6. Находит баг — оставляет комментарий в PR или заводит задачу.

### Сценарий 2: Регрессия перед релизом

QA смотрит git log за спринт:

```bash
git log --oneline --since="2 weeks ago"
```

Составляет список компонентов для регрессии на основе затронутых файлов.

### Сценарий 3: QA ведёт репозиторий с артефактами

QA создаёт репозиторий `qa-artifacts` и хранит там:

- чек-листы в формате `.md`;
- шаблоны баг-репортов;
- тест-кейсы;
- анализ API.

## 8. .gitignore

Файл `.gitignore` содержит список файлов и папок, которые Git не отслеживает:

```
node_modules/
.env
*.log
.DS_Store
```

**Почему QA это знает:** если конфигурационный файл с токенами или паролями случайно попал в репозиторий — это security-баг. Файлы `.env` не должны быть в репозитории никогда.

---

## 9. Markdown в репозиториях

Файлы `.md` используются для документации. GitHub автоматически рендерит их.

QA использует Markdown для:

- чек-листов;
- README репозитория;
- тест-кейсов;
- баг-репортов в текстовом формате.

Базовый синтаксис:

```markdown
# Заголовок 1
## Заголовок 2

**Жирный** *Курсив*

- Пункт списка

| Колонка 1 | Колонка 2 |
|---|---|
| Значение  | Значение  |

`код в строке`
```

## 10. Шпаргалка по командам для QA

| Команда | Когда использовать |
|---|---|
| `git clone <url>` | Первый раз получить репозиторий |
| `git pull` | Перед тестированием — обновить код |
| `git status` | Проверить состояние файлов |
| `git log --oneline` | Посмотреть историю коммитов |
| `git checkout <branch>` | Переключиться на ветку |
| `git branch -a` | Посмотреть все ветки |
| `git diff` | Посмотреть изменения в файлах |
| `git show <hash>` | Посмотреть конкретный коммит |

---

## 11. Типичные ошибки начинающего QA при работе с Git

- Тестирует не ту ветку (забыл переключиться).
- Работает с устаревшим кодом (забыл сделать `git pull`).
- Не читает историю коммитов перед регрессией.
- Не смотрит дифф в PR, а проходит по чек-листу без понимания, что изменилось.
- Хранит токены и пароли в репозитории.
