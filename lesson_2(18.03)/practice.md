# Lesson 2. Практика: Git и GitHub для тестировщика

## Цель практики

Этот файл — имитация реальных задач, с которыми junior QA сталкивается при работе с Git и GitHub.

После выполнения всех заданий ученик должен уметь:

- клонировать репозиторий и получать обновления;
- просматривать историю коммитов и диффы;
- переключаться между ветками;
- читать pull request как тестировщик;
- вести собственный репозиторий с тест-артефактами;
- объяснять, зачем конкретное действие с Git нужно QA.

---

## Как выполнять практику

По каждому заданию нужно отвечать не только **что сделал**, но и **зачем это нужно тестировщику**.

Слабый ответ:
- выполнил `git clone`;
- посмотрел `git log`.

Сильный ответ:
- выполнил `git clone`, чтобы получить актуальную версию репозитория перед тестированием;
- посмотрел `git log --oneline`, чтобы понять, что менялось за последние 2 недели и составить фокус-область регрессии.

## Блок 1. Установка и первый контакт с Git

### Задача

1. Убедись, что Git установлен. Выполни в терминале:

```bash
git --version
```

Ожидаемый результат: строка вида `git version 2.x.x`.

2. Настрой имя и email:

```bash
git config --global user.name "Твоё Имя"
git config --global user.email "твой@email.com"
```

3. Проверь конфигурацию:

```bash
git config --list
```

### Что сдать

- Скриншот вывода `git --version`;
- Скриншот вывода `git config --list` с именем и email.

---

## Блок 2. Клонирование публичного репозитория

### Сценарий как на проекте

Тебя добавили в команду. Нужно получить репозиторий, чтобы начать работу.

### Задача

Склонируй учебный публичный репозиторий:

```bash
git clone https://github.com/public-apis/public-apis.git
```

Затем:

1. Перейди в папку:

```bash
cd public-apis
```

2. Посмотри список файлов:

```bash
ls
```

3. Посмотри краткую историю коммитов:

```bash
git log --oneline -10
```

4. Посмотри текущую ветку и статус:

```bash
git status
git branch
```

### Что сдать

- Скриншот вывода `git log --oneline -10`;
- Краткое описание: что это за репозиторий, сколько файлов, какие последние 3 коммит-сообщения.

### Вопрос для размышления

Какую информацию из `git log` ты бы использовал, если бы тебе нужно было понять, что изменилось за последние 2 недели перед регрессионным тестированием?

## Блок 3. Работа с ветками

### Сценарий как на проекте

Разработчик открыл задачу. Нужно переключиться на его ветку и протестировать изменения.

### Задача

Создай новый локальный тестовый репозиторий:

```bash
mkdir qa-practice-repo
cd qa-practice-repo
git init
```

Создай файл:

```bash
echo "# QA Artifacts" > README.md
git add README.md
git commit -m "init: создан репозиторий QA артефактов"
```

Создай ветку для работы с чек-листом логина:

```bash
git checkout -b feature/login-checklist
```

Создай файл чек-листа:

```bash
echo "# Чек-лист: форма логина" > checklist-login.md
git add checklist-login.md
git commit -m "feat: добавлен чек-лист для формы логина"
```

Переключись обратно на `main`:

```bash
git checkout main
```

Убедись, что файл `checklist-login.md` в ветке `main` не существует:

```bash
ls
```

Переключись обратно на ветку:

```bash
git checkout feature/login-checklist
ls
```

### Что сдать

- Скриншоты переключения между ветками;
- Объяснение: зачем QA нужно переключаться на ветку разработчика перед тестированием.

## Блок 4. Создание собственного репозитория с тест-артефактами на GitHub

### Сценарий как на проекте

QA ведёт репозиторий с документацией: чек-листы, шаблоны баг-репортов, тест-кейсы.

### Задача

1. Зарегистрируйся на [github.com](https://github.com) (если ещё нет аккаунта).

2. Создай новый репозиторий:
   - Имя: `qa-artifacts`
   - Описание: `QA артефакты: чек-листы, баг-репорты, анализ API`
   - Тип: Public
   - Добавь README.md

3. Склонируй репозиторий на локальный компьютер:

```bash
git clone https://github.com/твой-username/qa-artifacts.git
cd qa-artifacts
```

4. Создай структуру папок:

```
qa-artifacts/
├── checklists/
│   └── login-checklist.md
├── bug-reports/
│   └── bug-report-template.md
└── api-analysis/
    └── reqres-users-analysis.md
```

5. Заполни `login-checklist.md` минимум 10 пунктами (можно взять из Урока 1).

6. Заполни `bug-report-template.md` шаблоном баг-репорта.

7. Сделай коммит и отправь на GitHub:

```bash
git add .
git commit -m "feat: добавлены начальные артефакты тестирования"
git push origin main
```

8. Открой репозиторий на GitHub и убедись, что все файлы появились.

### Что сдать

- Ссылку на репозиторий на GitHub;
- Скриншот страницы репозитория с файлами.

## Блок 5. Чтение и анализ Pull Request

### Сценарий как на проекте

Разработчик открыл PR. Тебе нужно его проревьюировать как QA.

### Задача

Открой любой публичный PR на GitHub. Можно использовать крупные публичные проекты:

- [github.com/public-apis/public-apis/pulls](https://github.com/public-apis/public-apis/pulls)
- [github.com/nicedoc/honkit/pulls](https://github.com/nicedoc/honkit/pulls)

Найди любой открытый или недавно закрытый PR и ответь на вопросы:

1. Какова цель изменения (что написано в описании PR)?
2. Сколько файлов затронуто?
3. Что именно добавлено (зелёные строки)?
4. Что именно удалено (красные строки)?
5. Есть ли комментарии к PR? Что в них обсуждают?
6. Как бы ты, как QA, сформулировал фокус-область тестирования на основании этого PR?

### Что сдать

- Ссылку на PR;
- Ответы на 6 вопросов выше;
- Список из 5 проверок, которые ты бы провёл после этого PR.

## Блок 6. История коммитов как инструмент регрессии

### Сценарий как на проекте

Завтра релиз. Нужно понять, что тестировать в первую очередь.

### Задача

Открой репозиторий на GitHub (можно использовать `public-apis` из Блока 2) и:

1. Перейди в раздел Commits.
2. Найди все коммиты за последние 2 недели.
3. Выпиши, какие компоненты или функции изменялись.
4. На основании этого составь список того, что нужно протестировать.

Затем сделай то же самое через терминал:

```bash
git log --oneline --since="2 weeks ago"
git log --oneline --since="2 weeks ago" --stat
```

Флаг `--stat` покажет, какие файлы были затронуты в каждом коммите.

### Что сдать

- Список из 5+ изменений с описанием, что именно менялось;
- Список из 5+ проверок, которые ты бы провёл на основании истории.

## Блок 7. .gitignore и безопасность

### Сценарий как на проекте

При ревью PR ты замечаешь, что разработчик случайно добавил в репозиторий файл `.env` с токенами API.

### Задача

1. В своём репозитории `qa-artifacts` создай файл `.env` с учебным содержимым:

```
API_TOKEN=test123
SECRET_KEY=mysecretkey
```

2. Проверь `git status`:

```bash
git status
```

Ты увидишь `.env` в списке неотслеживаемых файлов.

3. Создай файл `.gitignore`:

```bash
echo ".env" > .gitignore
echo "*.log" >> .gitignore
echo "node_modules/" >> .gitignore
```

4. Снова проверь `git status`:

```bash
git status
```

Файл `.env` больше не должен появляться.

5. Зафиксируй `.gitignore`:

```bash
git add .gitignore
git commit -m "chore: добавлен .gitignore"
git push
```

### Что сдать

- Скриншоты `git status` до и после добавления `.gitignore`;
- Объяснение: почему `.env`-файл нельзя коммитить в репозиторий, и как это связано с работой QA.

## Блок 8. Написание качественных коммит-сообщений

### Задача

Ниже приведены плохие коммит-сообщения. Перепиши каждое в хорошем формате:

| Плохое сообщение | Хорошее сообщение (напиши сам) |
|---|---|
| `fix` | |
| `update profile` | |
| `changes` | |
| `wip` | |
| `bug fixed` | |
| `added new stuff` | |

### Правила хорошего коммит-сообщения

- Начинается с типа: `feat:`, `fix:`, `test:`, `docs:`, `chore:`, `refactor:`
- Описывает суть изменения, а не процесс: `fix: исправлена валидация email` — хорошо; `fix: починил баг` — плохо.
- Короткое (до 72 символов).

### Что сдать

- Заполненную таблицу с переписанными сообщениями;
- Объяснение, почему качество коммит-сообщений важно для QA.

## Блок 9. Итоговое задание

### Задача

Создай в репозитории `qa-artifacts` структуру для хранения артефактов по Уроку 1:

```
qa-artifacts/
├── README.md
├── .gitignore
├── checklists/
│   ├── login-checklist.md
│   ├── user-profile-checklist.md
│   └── product-card-checklist.md
├── bug-reports/
│   ├── bug-report-template.md
│   ├── bug-001-price-display.md
│   └── bug-002-email-not-shown.md
└── api-analysis/
    ├── reqres-users.md
    ├── fakestoreapi-products.md
    └── dummyjson-users.md
```

Требования:

1. Каждый чек-лист должен содержать минимум 12 пунктов.
2. Каждый баг-репорт должен быть оформлен по шаблону из Урока 1.
3. Каждый файл анализа API должен содержать: обязательные поля, чувствительные поля, что отображать в UI.
4. Все изменения должны быть оформлены отдельными осмысленными коммитами.
5. Репозиторий должен быть опубликован на GitHub.

### Что сдать

- Ссылку на GitHub репозиторий;
- Скриншот истории коммитов (`git log --oneline`);
- Краткий отчёт: что сделал, зачем, что узнал нового.

---

## Критерии хорошего выполнения практики

Практика выполнена хорошо, если ученик:

- создал и опубликовал репозиторий на GitHub;
- коммит-сообщения осмысленные и соответствуют формату;
- структура папок логична и удобна для навигации;
- умеет объяснить, зачем каждый шаг нужен именно тестировщику;
- добавил `.gitignore` и понимает, зачем он нужен;
- умеет читать дифф и отвечать на вопрос: что нужно проверить после этих изменений.
